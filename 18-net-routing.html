אלגוריתמי ניתוב — DV ו-LS

איך ראוטרים בונים את הטבלאות שלהם — Bellman-Ford מבוזר ו-Dijkstra גלובלי.

טבלת ה-Forwarding בראוטר נבנית ע״י אלגוריתם ניתוב. שתי גישות מנוגדות: Distance Vector — כל ראוטר יודע רק את השכנים שלו ומעדכן איטרטיבית (Bellman-Ford); Link State — כל ראוטר יודע את כל הטופולוגיה ומריץ Dijkstra. ב-DV יש בעיית "ספירה לאינסוף" בבשורות רעות.

הצגת הבעיה

📘מסלול ועלות
  • מסלול: סדרת ראוטרים שההודעה עוברת בדרך מהמקור ליעד.
  • מסלול טוב: הכי פחות עלות. העלות יכולה להיות מספר ראוטרים, זמן השהיה, רוחב פס פנוי, או כסף.

הרשת מיוצגת כגרף: ראוטרים = קודקודים, קישורים = קשתות עם משקלים.

Distance Vector — וקטור מרחק

📘הרעיון

כל ראוטר מחזיק וקטור שאומר "כמה עולה לי להגיע לכל יעד". בצורה איטרטיבית מבוזרת:

  1. כל ראוטר שולח את הוקטור שלו לשכנים שלו.
  2. כל ראוטר מקבל וקטורים מהשכנים שלו ומעדכן את הוקטור שלו לפי Bellman-Ford.
  3. אם הוקטור השתנה — שולח לשכנים. אם לא — שקט.
  4. הרשת מתכנסת בסופו של דבר.
📐נוסחת Bellman-Ford

העלות מ-X ל-Y:

$$d_X(Y) = \min_{v \in \text{neighbors}(X)} \{ c(X, v) + d_v(Y) \}$$

אני בודק את כל השכנים שלי — אצל איזה מהם המסלול דרכי הזול ביותר ל-Y.

💡דוגמה — להגיע מ-U ל-Z

נניח שני שכנים של U: X ו-W.

  • עלות מ-U ל-X = 2. עלות מ-X ל-Z = 3. סך הכל דרך X: 5.
  • עלות מ-U ל-W = 4. עלות מ-W ל-Z = 3. סך הכל דרך W: 7.

אז U יעדיף לעבור דרך X. הוא ייעדכן בטבלה: Z ← דרך X, עלות 5.

בשורות טובות — הוספת ראוטר

📘תגובה לבשורות טובות

כשמתחבר ראוטר חדש E, כל הראוטרים צריכים לדעת איך להגיע אליו. התהליך:

  1. T=0: רק E מכיר את עצמו. כל השאר: מרחק = אינסוף.
  2. T=1: A מקבל וקטור מ-E (עלות 0 ל-E). A מעדכן: ‎d(E) = 1.
  3. T=2: B מקבל וקטור מ-A. B מעדכן: ‎d(E) = 2.
  4. T=3: C מקבל מ-B. d(E) = 3.
  5. T=4: D מקבל מ-C. d(E) = 4.

‎הרשת התכנסה ב-4 איטרציות — אורך המסלול. תגובה מצוינת.

בשורות רעות — בעיית "ספירה לאינסוף"

⚠הסרת ראוטר

נניח E התנתק. עכשיו אף אחד לא יכול להגיע אליו, אבל הראוטרים לא יודעים את זה.

מה קורה?

  1. A מאבד את החיבור ל-E. הוא יודע ש-d(E) = ∞ דרכו ישירות.
  2. אבל ‎B חושב ‎d(E) = 2 (זה היה נכון בעבר!). A רואה את הוקטור של B ומעדכן: ‎d(E) = 1 + 2 = 3.
  3. ‎B רואה ש-A התעדכן ל-3. עכשיו B חושב ‎d(E) = 1 + 3 = 4.
  4. ‎A רואה ‎d(E) של B = 4. מתעדכן ל-5. וכן הלאה...

הספירה עולה לאט לאט — לאינסוף. כל ראוטר מסתמך על שכן שמסתמך עליו. לולאת מידע גרועה.

📘הפתרון: הגדרת "אינסוף"

במקום לספור לאינסוף — מגדירים ערך מקסימלי. ב-RIP: 16.

כל מסלול שמגיע ל-16 נחשב "לא קיים". הראוטר מבין שהאובייקט לא בר-השגה.

המחיר

אי אפשר ליצור מסלולים שיוצרים יותר מ-15 קפיצות (hops). מגביל את גודל הרשת — אבל פותר את ספירה-לאינסוף.

Link State — לינק סטייט

📘הרעיון

במקום שכל ראוטר ידע רק את השכנים שלו — כל ראוטר יידע את כל הטופולוגיה. אחר כך כל אחד מריץ Dijkstra לוקאלית.

איך בונים את הטופולוגיה?

באמצעות Flooding: כל ראוטר שולח לכל הרשת מי השכנים שלו. כל ראוטר מקבל הצהרות שכנות מכל הרשת ובונה גרף.

אחרי שיש גרף

כל ראוטר מריץ Dijkstra מעצמו ומוצא את המסלול הקצר ביותר לכל יעד. בונה את הטבלה המקומית שלו.

📘למה Link State נמנע מספירה-לאינסוף

כשראוטר מתנתק — כולם מקבלים flooding תוך זמן קצר. ‎Dijkstra רץ מחדש על הגרף החדש (בלי הראוטר). מסלולים חדשים נמצאים תוך איטרציה אחת.

אין לולאת מידע — כי כל ראוטר יודע את הטופולוגיה בעצמו, לא מסתמך על שכן.

תרגיל

תרגיל בית 4 · שאלה 3.ד · ערך אינסוף ב-RIP

בפרוטוקול RIP נקבע ערך 16 עבור הגדרת המושג "אינסוף". למה יש צורך להגדיר מושג של אינסוף בפרוטוקול?

המוטיבציה — ספירה לאינסוף

RIP מבוסס על Distance Vector. כשראוטר ברשת מתנתק — נוצרת בעיית Count to Infinity:

  • שני ראוטרים מסתמכים זה על המידע של השני.
  • כל אחד מהם מעלה לאט לאט את המרחק.
  • אין סוף — הם ימשיכו ויעלו לעוד 100, 1000, 10,000, ...
הפתרון: ערך אינסוף מוגדר

במקום לחכות שהמרחק יגיע למילון ערכים — מגדירים סף קבוע. ב-RIP זה 16.

כשהמרחק מגיע ל-16, הראוטר מבין: "זה לא בר-השגה" ומסיר את הרשומה מהטבלה.

תוצאה: ספירת האינסוף מוגבלת — מסתיימת אחרי לכל היותר 16 איטרציות.

המחיר

‎RIP יכול לעבוד רק ברשתות שבהן המסלול הארוך ביותר < 16 קפיצות. ‎15 hops זה הגבול. ברשתות גדולות יותר — RIP לא מספיק.

ערך האינסוף נדרש כדי לעצור את ספירה-לאינסוף שמתרחשת בבשורות רעות (כשראוטר מתנתק). ‎16 הוא הסף ב-RIP. ברשתות גדולות יותר משתמשים ב-OSPF.
תרגול · הרצת Bellman-Ford

נתון הראוטר X עם שני שכנים: Y ו-Z. עלות מ-X ל-Y: 2. עלות מ-X ל-Z: 7. Y שלח את הוקטור שלו: {Y: 0, A: 3, B: 5}. Z שלח: {Z: 0, A: 1, B: 4}. מה הוקטור החדש של X?

הוקטור החדש של X

X לעצמו: 0.

X ל-Y: דרך Y ישירות = 2. (קצר יותר משום אפשרות אחרת.)

X ל-Z: דרך Z ישירות = 7. דרך Y: 2 + ? (Y לא הודיע על Z). אז 7.

X ל-A

דרך Y: ‎2 + 3 = 5

דרך Z: ‎7 + 1 = 8

מינימום: 5 דרך Y

X ל-B

דרך Y: ‎2 + 5 = 7

דרך Z: ‎7 + 4 = 11

מינימום: 7 דרך Y

הוקטור החדש של X: {X: 0, Y: 2, Z: 7, A: 5 (via Y), B: 7 (via Y)}.
📌סיכום הפרק
  • Routing = אלגוריתמים לבניית טבלת ה-Forwarding
  • Distance Vector: כל ראוטר יודע וקטור לכל יעד. ‎Bellman-Ford. ‎פרוטוקול: RIP
  • Link State: כל ראוטר יודע את כל הטופולוגיה. ‎Dijkstra. ‎פרוטוקול: OSPF
  • DV מגיב מצוין לבשורות טובות (תוך אורך המסלול)
  • DV סובל מ-ספירה-לאינסוף בבשורות רעות ← פתרון: סף ‎16 ב-RIP
  • LS לא סובל מהבעיה — Dijkstra רץ מחדש על כל גרף
  • LS יקר יותר בהודעות (flooding) — אבל יותר אמין
