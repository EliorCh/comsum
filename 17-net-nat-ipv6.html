NAT, IPv6, ו-ICMP

איך כל הבית שלך גולש דרך IP אחד · והעתיד של כתובות.

יש רק 4 מיליארד כתובות IPv4 — מעט מדי. ‎NAT פתר את הבעיה זמנית: כתובת פרטית פנימה, כתובת אחת ציבורית החוצה. ‎IPv6 הוא הפתרון לטווח ארוך — 128 ביטים. ‎ICMP הוא ה-"קונטרול פלין" של הרשת — איך ראוטרים מדווחים על בעיות.

NAT — Network Address Translation

‎NAT מסתיר רשת שלמה מאחורי IP ציבורי אחד. כל הלקוחות עם כתובות פרטיות 10.0.0.x.
📘הרעיון

הראוטר הביתי שלך הוא NAT. הוא:

  1. נותן לכל מחשב בבית כתובת פרטית (לדוגמה 10.0.0.5).
  2. לעצמו, מקבל מ-ISP כתובת ציבורית (138.76.29.7).
  3. כשמחשב פונה לאינטרנט — הראוטר מחליף את ה-IP מקור מהפרטי לציבורי.
  4. כשהתשובה חוזרת — הראוטר מחזיר את ה-IP יעד לפרטי הנכון.

איך NAT מתפעל — דוגמה מפורטת

💡תהליך מלא של בקשה ותגובה

שלב 1: PC1 שולח לעולם

‎PC1 (10.0.0.1, פורט 3345) שולח לשרת (128.119.40.186, פורט 80):

SRC IP : 10.0.0.1     SRC Port: 3345
DST IP : 128.119.40.186  DST Port: 80

שלב 2: ה-NAT מחליף ושומר בטבלה

הראוטר מחליף את ה-SRC IP והפורט:

SRC IP : 138.76.29.7   SRC Port: 5001  ← חדש!
DST IP : 128.119.40.186 DST Port: 80

שומר בטבלת NAT:

NAT sidePrivate side
138.76.29.7 : 500110.0.0.1 : 3345

שלב 3: השרת מגיב

SRC IP : 128.119.40.186 SRC Port: 80
DST IP : 138.76.29.7   DST Port: 5001

שלב 4: ה-NAT ממירה חזרה

הראוטר רואה DST Port=5001, מחפש בטבלה, מוצא ש-‎5001 ↔ ‎10.0.0.1:3345. מתרגם:

SRC IP : 128.119.40.186 SRC Port: 80
DST IP : 10.0.0.1      DST Port: 3345

ושולח ל-PC1 הפנימי.

המפתח: ה-NAT משתמש ב-פורט מקור (שונה לכל קשר) כדי לזהות לאיזה מחשב פנימי להחזיר.

יתרונות NAT

📘למה NAT הצליח?
  1. חוסך כתובות: כל הבית/חברה מהאינטרנט נראים כ-IP אחד.
  2. גמישות פנימית: אפשר לשנות כתובות פנימיות בלי לעדכן את העולם. אם מעבירים ISP — רק ה-NAT מקבל IP חדש.
  3. אבטחה: מחשב מבחוץ לא יכול ליזום קשר למחשב פנימי — ה-NAT לא יידע אליו להעביר. דמוי-Firewall.

חסרונות NAT

⚠הצד האפל של NAT
  • שובר עקרון השכבות: ראוטר אמור לטפל רק עד שכבת הרשת (3). NAT נוגע ב-port (שכבה 4). הפרת ההפרדה.
  • אי אפשר ליזום קשר פנימה: אם רוצים לארח שרת בבית — בעיה. ה-NAT לא יודע למי להעביר.
  • בעיות עם פרוטוקולים מסוימים: פרוטוקולים שמשתמשים ב-IP בתוך ה-payload (FTP, SIP) לא עובדים נכון.

‎3 פתרונות לקבלת קשר נכנס

  1. Port Forwarding סטטי: הגדרה בראוטר — "כל מה שמגיע לפורט 2300 ← תעביר ל-IP פנימי X".
  2. DynDNS: שירות שעוקב אחרי ה-IP של ה-NAT שלך וממפה לדומיין קבוע (myname.dyndns.com).
  3. איש קשר באמצע: שני המחשבים יוזמים קשר לשרת מרכזי (Skype, WhatsApp), והוא מקשר ביניהם.

IPv6 — הפתרון הסופי

📘הרעיון: כתובות יותר ארוכות

‎128 ביטים במקום 32. גודל הכתובת ב-IPv6:

$$2^{128} \approx 3.4 \times 10^{38} \text{ כתובות}$$

זה ‎10²⁸ כתובות לכל אדם בעולם. לא ייגמרו.

פורמט כתיב

‎8 קבוצות של 16 ביטים כל אחת, מופרדות ב:נקודותיים, בהקסה:

2001:0db8:85a3:0000:0000:8a2e:0370:7334

אפשר לקצר רצף 0-ים ל-:: (פעם אחת בכתובת):

2001:db8:85a3::8a2e:370:7334

📘שיפורים נוספים ב-IPv6

הדר באורך קבוע (40 בתים)

ב-IPv4 ההדר באורך משתנה (בגלל options). בכל ראוטר צריך לבדוק כמה גדול. ב-IPv6 — תמיד 40 בתים. ראוטרים מהירים יותר.

אין פרגמנטציה בראוטרים

אם דאטהגרם גדול מ-MTU — הראוטר זורק ושולח ICMP לשולח שיקטין. השולח מבצע Path MTU Discovery ושולח דאטהגרמים בגודל הנכון מההתחלה.

אין Checksum בהדר

נשמטה כי שכבות אחרות (תעבורה, לינק) כבר עושות checksum. חיסכון בזמן עיבוד בכל ראוטר.

שדה Hop Limit במקום TTL

בדיוק אותה פונקציה — נמנע מלולאות.

מעבר מ-IPv4 ל-IPv6 — Tunneling

⚠למה זה לא קל?

לא ניתן להחליף את כל הראוטרים בעולם בו-זמנית. רובם תומכים ב-IPv4 בלבד. אז יש תקופת מעבר ארוכה.

ראוטרים תומכי-IPv6 גם תומכים ב-IPv4 (תאימות לאחור). אבל לא הפוך.

📘Tunneling — פתרון המעבר

אם ראוטר IPv6 צריך לשלוח דרך ראוטר IPv4 — הוא עוטף את הדאטהגרם IPv6 בתוך דאטהגרם IPv4.

IPv4 הדר | IPv6 הדר | TCP/UDP segment | Data

הראוטר IPv4 מטפל בזה כמו דאטהגרם רגיל. הראוטר IPv6 בצד השני פותח את העטיפה ומקבל את ה-IPv6 המקורי.

זה דמוי "מנהרה" — הדאטהגרם IPv6 עובר דרך תשתית IPv4.

ICMP — Internet Control Message Protocol

📘למה צריך ICMP?

ראוטרים זורקים פאקטות ב-best effort בלי דיווח. אבל לפעמים אנחנו רוצים לדעת למה. ICMP הוא הפרוטוקול שמדווח על שגיאות ושאלות.

  • איפה רץ: אפליקציה ברמת הראוטר.
  • שכבה: שכבת הרשת (3) — בצמוד ל-IP.
  • מעל IP: ICMP messages עטופים ב-IP datagram עם Protocol=1.
TypeCodeמשמעות
00echo reply (תגובה ל-ping)
30dest network unreachable
31dest host unreachable
32dest protocol unreachable
33dest port unreachable
34fragmentation needed but DF set
80echo request (ping)
110TTL expired
📘שימושים נפוצים

ping

שולח Type 8 (echo request) ליעד. אם היעד פעיל — מחזיר Type 0 (echo reply). מודד RTT.

traceroute

חכם יותר. שולח חבילות עם TTL הולך וגדל: TTL=1, TTL=2, וכו'. כל ראוטר שמקבל TTL=0 זורק ושולח ICMP Type 11 (TTL expired) חזרה לשולח. ככה מגלים את כל הראוטרים במסלול.

תרגיל

תרגול · NAT

‎PC ברשת ביתית עם IP פנימי 192.168.1.5 שולח בקשה לשרת 8.8.8.8 בפורט 53 (DNS). ה-NAT בכתובת ציבורית 203.0.113.7.

  1. מה ה-PC שולח בפועל (לפני NAT)?
  2. מה ה-NAT שולח לאינטרנט (אחרי המרה)?
  3. מה השרת מקבל ומגיב אליו?
  4. איך ה-NAT יודע למי להחזיר?
1. ה-PC שולח
SRC IP : 192.168.1.5    SRC Port: 49234 (אקראי)
DST IP : 8.8.8.8        DST Port: 53
2. ה-NAT ממיר

ה-NAT מחליף את ה-SRC IP ל-IP שלו, ואת ה-SRC Port לפורט חדש (לשמירה בטבלה):

SRC IP : 203.0.113.7    SRC Port: 7892 (חדש)
DST IP : 8.8.8.8        DST Port: 53

ושומר בטבלה: 203.0.113.7:7892 ↔ 192.168.1.5:49234

3. השרת מגיב
SRC IP : 8.8.8.8        SRC Port: 53
DST IP : 203.0.113.7    DST Port: 7892

השרת לא יודע על ה-NAT — הוא רואה את התשובה הולכת ל-203.0.113.7.

4. ה-NAT מחזיר ל-PC

ה-NAT מקבל פאקטה עם DST Port=7892. מחפש בטבלה — זה משויך ל-192.168.1.5:49234.

SRC IP : 8.8.8.8        SRC Port: 53
DST IP : 192.168.1.5    DST Port: 49234

שולח ל-PC הפנימי. ה-PC לא יודע שהיה NAT באמצע — הוא רואה תגובה רגילה משרת 8.8.8.8.

ה-NAT מתפקד כשכבת תרגום שקופה. שני הצדדים רואים מציאות שונה — ה-PC רואה את עצמו עם 192.168.1.5, השרת רואה אותו כ-203.0.113.7. ה-Port הוא המפתח לזיהוי.
📌סיכום הפרק
  • NAT: רשת שלמה פנימה ‎= IP אחד החוצה. ‎חוסך כתובות, פוגע בעקרון השכבות.
  • NAT משתמש ב-Port מקור לזיהוי איזה לקוח פנימי שייך לאיזה תקשורת.
  • בעיה: אי אפשר ליזום קשר נכנס. ‎פתרונות: Port Forwarding, DynDNS, איש קשר באמצע.
  • IPv6: 128 ביטים, הדר קבוע 40 בתים, בלי פרגמנטציה ובלי checksum בהדר
  • מעבר: Tunneling — לעטוף IPv6 בתוך IPv4
  • ICMP: פרוטוקול לדיווחי שגיאה ובקרה. ‎Type+Code לכל סוג הודעה.
  • שימושים מפורסמים: ping (Echo Req/Reply) ו-traceroute (TTL expired)
