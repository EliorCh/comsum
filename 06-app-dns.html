DNS — מערכת שמות הדומיינים

איך www.amazon.com הופך ל-‎52.94.236.248 — מסע דרך עץ עולמי של שרתים.

DNS = עץ. Root בראש. תחתיו TLD (com, org, edu...). תחתיהם Authoritative (amazon.com).

DNS הופך שמות אינטרנט (www.amazon.com) ל-IP. הוא מיושם כ-בסיס נתונים מבוזר בעולם בצורת עץ. למבחן: לדעת את ההיררכיה (Root → TLD → Authoritative), ההבדל בין שאילתה איטרטיבית לרקורסיבית, ו-סוגי הרשומות.

למה DNS?

📘הבעיה

מחשב מזוהה ע״י IP (32 ביטים ב-IPv4). יש שתי בעיות:

  • IP זה לא נוח לבני אדם — אף אחד לא יזכור 52.94.236.248.
  • IP יכול להשתנות — שרת עובר מ-data center אחד לשני, ה-IP משתנה אבל אנחנו רוצים שהשם יישאר.

הפתרון: מערכת שתתרגם שם נוח (כמו www.amazon.com) לכתובת IP.

📘DNS — סקירה
  • אפליקציה בשכבת האפליקציה — לא חלק מליבת הרשת.
  • בסיס נתונים מבוזר בעולם, מבוסס על שרתי שמות.
  • מסודר בהיררכיה (עץ).
  • אם DNS קורס — הרשת ממשיכה לעבוד, רק שמות לא נפתרים. (אם יש לך כבר IP — אפשר להמשיך).
  • תומך באלייסים (שמות נרדפים) — דרך רשומות CNAME.
  • פורט DNS: 53. רץ מעל UDP (לבקשות קצרות).

היררכיית שרתי DNS

📘3 רמות

‎1. Root Name Server

  • שרת השורש של ההיררכיה.
  • נקודת ההתחלה של כל חיפוש (אם אין במטמון).
  • כל שרת DNS מכיר את ה-IP שלו.
  • 13 שרתים בעולם (אבל כל אחד עם הרבה רפליקות).
  • לא יודע IP-ים סופיים — יודע רק איפה למצוא את ה-TLDs.

2. TLD — Top Level Domain

  • שרתים שמטפלים בסיומות: .com, .org, .edu, .gov, .il, .de...
  • מנוהלים ע״י ארגונים גדולים (Verisign מנהלת את .com).
  • לא יודעים IP סופי — יודעים איפה לחפש את ה-Authoritative.

‎3. Authoritative Name Server

  • השרת שבאמת יודע את ה-IP של דומיין מסוים.
  • מתוחזק ע״י בעלי הדומיין (אמאזון מתחזקת את amazon.com).
  • מחזיק את כל רשומות הדומיין: www.amazon.com, shop.amazon.com, וכו'.
⚠Local DNS Server — חוץ מההיררכיה

Local Name Server אינו חלק מההיררכיה הפורמלית. הוא:

  • שרת DNS שמסופק ע״י ה-ISP שלך (או שאתה מגדיר ידנית, כמו ‎8.8.8.8 של גוגל).
  • מתפקד כפרוקסי לשמות — אם מישהו ברשת שלך כבר שאל את אותה שאלה, התשובה במטמון של ה-Local.
  • פנייה ראשונה של המחשב שלך תמיד תהיה אליו.
  • אם הוא לא יודע — הוא יפנה לשאר ההיררכיה.

‎2 סוגי שאילתות — איטרטיבי ורקורסיבי

איטרטיבי: Local DNS שואל כל שרת בעצמו. כל שרת מצביע על הבא, לא ממשיך לבד.
📘שאילתה איטרטיבית (Iterative)

ה-Local DNS עושה את כל החיפוש. כל שרת ב-Hierarchy רק מצביע על השרת הבא — לא מבצע חיפוש בעצמו.

שלבים (לדומיין gaia.cs.umass.edu)

  1. הלקוח (cis.poly) שואל את ה-Local DNS.
  2. Local DNS שואל את ה-Root.
  3. Root עונה: "שאל את ה-TLD של .edu".
  4. Local DNS שואל את TLD .edu.
  5. TLD עונה: "שאל את Authoritative של umass.edu".
  6. Local DNS שואל את ה-Auth.
  7. Auth מחזיר את ה-IP.
  8. Local DNS מחזיר את ה-IP ללקוח.

היתרון: השרתים בהיררכיה (Root, TLD) לא בעומס — הם רק נותנים תשובה קצרה ולא ממשיכים.

📘שאילתה רקורסיבית (Recursive)

במקום שה-Local DNS יעבור שרת-שרת — הוא שואל את ה-Root, וה-Root ממשיך בעצמו: שואל את ה-TLD, שמשאל את ה-Auth, והתשובה חוזרת בשרשרת.

תהליך

  1. Local DNS שואל את ה-Root: "מה ה-IP של gaia.cs.umass.edu?"
  2. Root שואל את TLD .edu (במקום להפנות).
  3. TLD שואל את ה-Auth.
  4. Auth מחזיר ל-TLD.
  5. TLD מחזיר ל-Root.
  6. Root מחזיר ל-Local.
  7. Local מחזיר ללקוח.

החיסרון: ה-Root וה-TLD נכנסים לעומס גדול — הם עושים יותר עבודה.

בעולם האמיתי: Root ו-TLD מסרבים לרקורסיה. רק Local DNS בדרך כלל מאפשר אותה.

למה איטרטיבי עדיף בעולם? כי הוא מקל על שרתי השורש. אם כל בקשה הייתה רקורסיבית — Root ו-TLD היו קורסים. איטרטיבי מפזר את העומס.

Caching ב-DNS

📘שמירת תשובות

כששרת DNS מקבל תשובה — הוא שומר אותה במטמון לפעמים הבאות.

לכל רשומה יש TTL — Time To Live בשניות. אחרי שה-TTL פוקע, השרת מוחק את המטמון של הרשומה הזו.

יתרונות

  • חיפוש מהיר יותר.
  • פחות תנועה באינטרנט.
  • פחות עומס על Root ו-TLD (דומיינים נפוצים תמיד במטמון של Local DNS).

חיסרון

אם IP של שרת השתנה — המטמון לא יודע. ה-TTL נועד לטפל בזה — בעלי דומיינים מגדירים TTL קצר אם הם מצפים לשינויים.

סוגי רשומות (Resource Records)

TypeמטרהNameValue
A מיפוי שם דומיין ל-IP example.com ‎93.129.0.24
CNAME שם נרדף (alias) www.example.com example.com (השם האמיתי)
NS מי השרת המנהל של הדומיין example.com dns1.example.com
MX לאן לשלוח מיילים של הדומיין example.com mail.example.com
📘מבנה הודעת DNS

שאילתה ותגובה משתמשות באותו פורמט (חוסך עיבוד אצל השרת).

Header (12 בתים)

  • Identification (16 ביט): ‎ID של השאילתה — כך התגובה תוכל להיות מקושרת לבקשה הנכונה.
  • Flags: שאילתה/תגובה, האם רקורסיבי, האם authoritative.
  • # questions, # answers, # authority, # additional: כמה רשומות בכל חלק.

Body

  • Questions: מה הלקוח שואל.
  • Answers: רשומות התשובה.
  • Authority: מי המנהל.
  • Additional: מידע שהשרת חושב שיועיל.

אבטחת DNS

⚠איומים

DDoS על Root

הצפת שרתי Root. עד עכשיו לא הצליח — מוגן ע״י סינון תעבורה. גם אם יצליח חלקית, רוב הדומיינים הנפוצים במטמון של ה-Locals.

DDoS על TLD

מסוכן יותר. אם TLD .com יורד, אתרים נפוצים יכולים להפוך לבלתי נגישים.

DNS Spoofing

תוקף מחזיר תגובות זדוניות לשאילתות DNS. אם הקש שומר אותה — כל מי שישאל יקבל IP זדוני. הפתרון: DNSSEC — חתימה דיגיטלית על תגובות.

תרגילים

תרגיל בית 1 · שאלה 2.1

מנו שלוש סיבות עיקריות לכך שאין שרת DNS אחד עולמי. (גודל, קרבה, ושרידות survivability).

‎1. גודל (Size)

יש מיליארדי דומיינים רשומים בעולם. שרת אחד לא יוכל להחזיק את הטבלה הענקית הזו. גם אם יוכל — חיפוש בה יהיה איטי מאוד.

‎2. קרבה לאזורים שונים

אם שרת ה-DNS היחיד בארה״ב, כל בקשה מישראל תעבור את האוקיינוס — פרופגיישן גבוה, חוויה איטית. פיזור גיאוגרפי מקרב את ה-DNS למשתמשים.

‎3. שרידות (Survivability)

אם השרת היחיד יורד — כל האינטרנט מפסיק לעבוד (שמות לא נפתרים). שרת בודד הוא Single Point of Failure. ‎13 שרתי Root + רפליקות = עמידות לקריסות.

‎3 סיבות: גודל, קרבה גאוגרפית, ושרידות. כולן מצדיקות מערכת מבוזרת.
תרגיל בית 1 · שאלה 2.2 · ממבחן 2010

סטודנט נדרש לכתוב פונקצית שירות של ‎OS עבור דפדפנים — מקבלת URL ומחזירה IP. נתון שאין תמיכה בשאילתות רקורסיביות בשרת ה-DNS המקומי, אך יש תמיכה רקורסיבית בשרתי DNS אחרים. תארו מימוש אפשרי.

המגבלה

ה-Local DNS לא יודע לעשות רקורסיה. אם נשאל אותו על gaia.cs.umass.edu והוא לא יודע, הוא יחזיר רק "תפנה לשרת אחר" ולא ימשיך.

המימוש

הפונקציה תבצע שאילתה איטרטיבית בעצמה:

  1. שואלת את ה-Local DNS. אם יש לו תשובה (במטמון) — מחזירה.
  2. אם לא — הלוקאל יחזיר "שאל את ה-Root" עם ה-IP שלו.
  3. הפונקציה שולחת שאילתה ל-Root. ה-Root יחזיר "שאל את ה-TLD" (לדוגמה .edu).
  4. הפונקציה שולחת ל-TLD. הוא יחזיר "שאל את ‎umass.edu Auth".
  5. הפונקציה שולחת ל-Auth. הוא יחזיר את ה-IP.
  6. מחזירים ל-Browser.
חלופה — תשתמש בשרת רקורסיבי אחר

אם ב-OS מוגדר שרת רקורסיבי חיצוני (לדוגמה ‎8.8.8.8 של גוגל), פשוט נפנה אליו — הוא יעשה את כל העבודה ויחזיר IP.

או הפונקציה עושה איטרציה בעצמה, או היא מפנה לשרת DNS רקורסיבי חיצוני (כמו DNS פומבי).
תרגיל בית 1 · שאלה 2.3

הסבירו בקצרה מה הם: local name server, root name server, authoritative name server. נמקו למה תהליך שאילתה איטרטיבי עדיף על רקורסיבי.

‎3 סוגי השרתים
  • Local Name Server: שרת DNS שמספק ה-ISP שלך. הפנייה הראשונה של המחשב. לא חלק מההיררכיה הפורמלית — מתפקד כפרוקסי לשמות.
  • Root Name Server: שרת השורש של ההיררכיה. 13 שרתים בעולם (עם רפליקות). יודע איפה ה-TLDs.
  • Authoritative Name Server: השרת שאחראי על דומיין ספציפי. יודע את ה-IP הסופי. מתוחזק ע״י בעלי הדומיין.
למה איטרטיבי עדיף

איטרטיבי מקל על שרתי השורש וה-TLD:

  • ב-Iterative: Root מקבל שאלה, נותן תשובה קצרה, ושכח.
  • ב-Recursive: Root מקבל שאלה, ממשיך לשאול בעצמו את TLD, ממתין, מקבל תשובה, מעביר חזרה. הוא חייב לעבוד יותר ולזכור.

בהתחשב שיש טריליוני בקשות DNS ביום, איטרטיבי הוא היחיד שיכול לשרוד.

איטרטיבי = פחות עומס על שרתי הליבה. הוא מקל על Root ו-TLD, וזו הסיבה שהמערכת בעולם בנויה ככה.
סיכום למבחן · נקודות עיקריות

בחר את ההצהרות הנכונות לגבי DNS:

  1. מערך שרתי ה-DNS בעולם הוא מבוזר
  2. מערך שרתי ה-DNS הוא סדרתי
  3. שרתי ה-DNS שייכים לגופים פרטיים בלבד
  4. אם DNS קורס — כל האינטרנט מפסיק לעבוד
בדיקה משפט-משפט
  • 1. מבוזר ✓ נכון. מיליוני שרתי DNS בעולם, מסודרים בעץ. לא מרכז אחד.
  • 2. סדרתי ✗ לא נכון. ה-DNS הוא היררכי ולא סדרתי. כל רמה ממליצה על הבאה — לא רץ ברצף לינארי.
  • 3. גופים פרטיים בלבד ✗ לא נכון. Root ו-TLD מנוהלים ע״י ארגונים בינלאומיים (ICANN, Verisign). Authoritative יכולים להיות פרטיים, אבל לא רק.
  • 4. אינטרנט נופל ✗ לא נכון. אם DNS נופל, רק תרגום שמות נפסק. מי שיודע IP יכול עדיין להגיע. הליבה ממשיכה לעבוד.
רק הצהרה 1 נכונה. מבוזר, היררכי (לא סדרתי), משולב פרטי/ציבורי, לא Single Point of Failure.

‎4 סוגי העברת נתונים ברשת

📘קסטים

הסיכום למבחן מציין ‎4 סוגי תעבורה — חשוב להכיר את השמות:

  • Unicast (יוניקאסט): ‎שולח יחיד למקבל יחיד. ‎המקרה הרגיל — רוב התעבורה ברשת.
  • Broadcast (ברודקאסט): ‎שולח יחיד לכולם (בסאבנט). ‎כתובת FF:FF:FF... ב-MAC, או 255.255.255.255 ב-IP. ‎דוגמה: DHCP Discover.
  • Multicast (מולטיקאסט): ‎שולח יחיד לקבוצה של מקבלים (subset). ‎דוגמאות: סטרימינג, פגישות וידאו.
  • Anycast (אניקאסט): ‎שולח יחיד לקרוב ביותר מתוך קבוצת מקבלים עם אותה כתובת. ‎דוגמה: שרתי Root של DNS — אותה כתובת IP, מספר שרתים, ההודעה הולכת לקרוב ביותר גאוגרפית.
📌סיכום הפרק
  • DNS = שירות לתרגום שם דומיין ← IP. אפליקציה, לא חלק מהליבה.
  • ‎3 רמות היררכיה: Root → TLD → Authoritative
  • Local DNS מחוץ להיררכיה — פרוקסי לשמות, נקודת פנייה ראשונה
  • ‎2 סוגי שאילתות: איטרטיבי (Local שואל בעצמו את כולם) ורקורסיבי (כל שרת ממשיך לבד)
  • איטרטיבי עדיף — מקל על Root ו-TLD
  • ‎4 סוגי רשומות: A (IP), CNAME (alias), NS (name server), MX (mail)
  • Cache עם TTL — חוסך זמן ועומס; פוקע לפי TTL
  • DNS רץ מעל UDP על פורט 53
  • איומים: DDoS, DNS Spoofing. הגנה: סינון, DNSSEC
